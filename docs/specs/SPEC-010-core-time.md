## 1. 目的

現実の人生では「遊びの時間」はつねに **学びごと・仕事** とセットである。  
本作では学び／仕事を **遊びを妨げる時間ではなく、新しい遊びを発見するための時間** として扱う。

コアタイムの役割：
1. **時間的な制約**：その時間は遊びを「選べない」。プレイヤーは受動的に過ごす。
2. **遊びの発見装置**：コアタイムを過ごすことで、新しい遊びが解禁される。
3. **パラメーター成長装置**：コアタイムにも原体験・友人数・ジョウネツが加算される。
4. **ジレンマ設計の強化**：時間とカネと体力のトレードオフが、ライフステージで変質する。

## 2. 用語定義

- **ライフステージ（Life Stage）**：年齢帯ごとの人生フェーズ。
- **コアタイム（Core Time）**：そのライフステージで固定的に拘束される時間帯。
- **コアタイム枠（Core Time Slot）**：1日の中でコアタイムに該当する `[startHour, endHour]` の区間。
- **コアタイム消化（Core Time Consumption）**：時刻がコアタイム枠を跨ぐ際に、自動でイベントや原体験加算を発生させる処理。
- **発見（Discovery）**：コアタイムを通じて、新しい遊びや友人・パラメーターが解禁されるイベント。

## 3. ライフステージとコアタイム マトリクス

| 年齢 | ライフステージ | コアタイム | 特徴 | 個別仕様 |
|---|---|---|---|---|
| 1〜3 | 保育園 | 09:00〜16:00 | 曜日によって室内遊び/公園遊び（13:00〜15:00） | [SPEC-011](./SPEC-011-nursery.md) |
| 4〜6 | 幼稚園 | 09:00〜14:00 | 友人が増える／基礎パラメーターが増える | [SPEC-012](./SPEC-012-kindergarten.md) |
| 7〜12 | 小学校 | 08:00〜15:30 | 様々な施設訪問で発見／遊びの幅が爆発的に広がる | [SPEC-013](./SPEC-013-elementary.md) |
| 13〜15 | 中学校 | 08:00〜17:00 | 部活動で友人増加／専門的な遊びの解禁 | [SPEC-014](./SPEC-014-juniorhigh.md) |
| 16〜18 | 高校 | 09:00〜16:00 | 進路選択・校外活動／バイト解禁の準備 | [SPEC-015](./SPEC-015-highschool.md) |
| 19〜22 | 大学 | 09:00〜18:00 | **コアタイムを自由に決められる** | [SPEC-016](./SPEC-016-university.md) |
| 23〜65 | 社会人 | 09:00〜18:00 | 仕事を通じた出会い／カネの余剰は増えるが時間が削られる | [SPEC-017](./SPEC-017-worker.md) |

> 各ライフステージの詳細な挙動・発見ロジック・解禁遊びは、上表の個別仕様を参照。  
> 本仕様書は **フレームワーク（共通の骨組み）** のみを定義する。

## 4. 共通データモデル

### 4.1 ライフステージマスタ

```ts
type LifeStage = {
  id: "nursery" | "kindergarten" | "elementary" | "juniorhigh" | "highschool" | "university" | "worker" | "retirement";
  label: string;          // 例 "保育園"
  ageMin: number;         // 開始年齢（含む）
  ageMax: number;         // 終了年齢（含む）
  coreTime: CoreTimeSlot | null;  // 老後は null
  customizable?: boolean; // 大学=true
};
```

### 4.2 コアタイム枠

```ts
type CoreTimeSlot = {
  startHour: number;       // 9 = 09:00
  endHour: number;         // 16 = 16:00
  label: string;           // HUD に表示する短い名前（例 "保育園"）
  weekdaysOnly?: boolean;  // true なら土日は発生しない（プロト段階ではすべて true 想定だが未使用）
};
```

## 5. 挙動（Behavior）

### 5.1 コアタイムの解決
```
function resolveCoreTime(age):
  対応する LifeStage を年齢で特定
  その LifeStage.coreTime を返す（null なら老後 = コアタイム無し）
```

### 5.2 コアタイム消化のタイミング
**起床画面（S1）から遊び選択画面（S2）へ進む直前**に、`player.clockHour` がコアタイムの `startHour` 以前であれば、**コアタイム画面（S6）を挿入** する。

画面遷移の流れ：
```
S1 起床
  ↓ 今日を始める
[判定] 今日コアタイムがあるか？
  ├─ ある → S6 コアタイム画面（各ステージ固有）
  │          ↓ コアタイム完了
  │         S2 遊びを選ぶ（時刻は endHour に進んでいる）
  └─ ない → S2 遊びを選ぶ
```

### 5.3 時計の進行
- コアタイム消化時：時刻を `coreTime.endHour` に進める。
- 余剰時間：`spareHours = (就寝時刻 − endHour − 食事・通学時間)` として再計算。

> 注：プロト段階では、起床時 `spareHours = 6h` を固定値で与える代わりに、コアタイム消化後に **動的に再計算** するモードに切り替える。

### 5.4 コアタイム中の処理（共通）
コアタイム画面（S6）で、以下の共通処理を必ず行う：
1. 時刻の進行（`startHour` → `endHour`）
2. 体力消費（ステージごとに既定値。SPEC-011 以降で定義）
3. **発見（Discovery）判定**：一定確率でイベントを発火し、
   - 新しい遊びを解禁
   - 友人数を増やす
   - 基礎パラメーターを加算
4. コアタイムに応じた **固有の経験値** を加算（例：保育園では「身体+5, 社会+3」）

### 5.5 発見（Discovery）
各ライフステージに `discoveryPool` を定義する：
```ts
type Discovery = {
  id: string;
  text: string;         // 画面に表示する発見テキスト
  weight: number;       // 抽選重み
  unlockPlayId?: string;// この発見で解禁される遊びID
  gain?: Partial<PlayerGain>; // 経験値・友人数などの加算
};
```
プロト段階では **日ごとに 0 or 1 件発火**（確率 60%）。

### 5.6 大学のカスタマイズ（将来）
大学ライフステージ（SPEC-016）のみ `customizable: true`。  
プレイヤーがコアタイム枠を画面から変更できる。**本仕様書では枠組みのみ定義。詳細は SPEC-016 にて。**

### 5.7 コアタイムのスキップ／回避
- **原則スキップ不可**。コアタイムは人生の一部として受動的に経過する。
- 例外として、発見イベントには選択肢を与える場合がある（例：幼稚園で「A を学ぶ / B を学ぶ」）。
  この場合も時間そのものはスキップできず、`endHour` に進む。

## 6. UIへの反映

### 6.1 HUD
- ライフステージ名を HUD に表示（例「🏫 保育園」）。
- コアタイムが進行中の間は「🔒 保育園中」のような警告アイコン（プロト段階では省略可）。

### 6.2 新画面 S6 コアタイム
- ライフステージ共通のテンプレート画面を用意し、`label`、描写文、発見イベントを差し替えて表示する。
- 画面設計書：`docs/screen-design.md` §2.6（追加）

### 6.3 遊び選択画面（S2）への影響
- コアタイム消化後の S2 では、余剰時間がコアタイム反映後の値になる。
- 発見で解禁された遊びがあれば、先頭に「✨ 新しく覚えた！」バッジ付きで表示（将来の拡張）。

## 7. 想定される利用シナリオ

### 7.1 保育園ステージ（SPEC-011）
- 2歳。朝 06:30 起床 → 今日を始める → **S6 保育園画面** → 「公園で滑り台の遊び方を教えてもらった」発見 → 16:00 で帰宅 → S2 遊びを選ぶ。

### 7.2 社会人ステージ（SPEC-017、将来）
- 30歳。07:00 起床 → S6 仕事画面 → 「取引先と飲みに行った」発見で友人+1 → 18:00 終業 → 余剰時間2hで遊びを選ぶ。

## 8. 例外・エッジケース

- 老後（66歳以降）は `coreTime = null` のため、S6 はスキップされる（従来の S1→S2 動線）。
- 体調不良（SPEC-006）でコアタイムを休む機能は将来検討（プロトでは非対応）。
- 休日（土日）の扱い：プロト段階ではすべての日でコアタイムが発生する（将来 `weekdaysOnly` を導入）。
