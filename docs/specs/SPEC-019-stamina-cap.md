# SPEC-019: 体力上限（年齢依存）と体力ゼロ挙動

| 項目 | 内容 |
|---|---|
| 仕様ID | SPEC-019 |
| 機能名 | 体力上限（年齢依存）／体力ゼロ時の強制睡眠・強制終了 |
| 対応ファイル | `prototype/game.js`（`STAMINA_CAP_TABLE`, `staminaCapForAge`, `handleStaminaDepleted`） |
| 関連仕様 | SPEC-001, SPEC-006, SPEC-008, SPEC-010, SPEC-011, SPEC-018, SPEC-020 |
| ステータス | Active（保育園分のみ実装）／他ライフステージはフレームワーク定義のみ |
| 最終更新 | 2026-04-18 |

## 1. 目的

「遊ぶにも体力がいる」という人生のリアリティをシミュレートする。  
年齢によって体力の器は大きく変動し、**幼少期は少ない・青年期にピーク・老後は緩やかに減衰** する曲線を描く。  
また、体力ゼロ時の挙動をライフステージ別に定義することで、**「遊び過ぎ」のリスクとリターン** を体感させる。

## 2. 用語定義

- **体力上限（Stamina Cap）**：`player.stamina` の最大値。年齢に応じて変動する。
- **体力ベース上限**：年齢から決まる標準値（本仕様書 §5.1）。
- **体力ボーナス上限**：遊びや発見イベントで伸びる追加上限（例：スポーツ系を続けると +20 など）。
- **強制睡眠**：体力ゼロで発動。2時間の仮眠を挿入し時刻を進める。
- **強制終了**：体力ゼロで発動。その日は終了し即就寝。翌朝の余剰時間が失われる。
- **体力ゼロ記録日**：その日に 1 回でも体力が 0 に触れた日。`player.depletedDays` に日付を記録する。

## 3. 入力（Input）

- プレイヤー年齢 `player.age`
- 年齢→体力上限のテーブル（§5.1）
- 遊びまたはコアタイムの `staminaCost`
- 遊びの `staminaBonus`（遊びによる体力上限の恒久ボーナス、将来拡張）

## 4. 出力（Output）

- `player.staminaCap`（動的に再計算される体力上限）
- `player.stamina`（現在値。`staminaCap` でクランプされる）
- 体力ゼロ時のフロー分岐（強制睡眠 or 強制終了）
- 老後の翌年減衰緩和フラグ `player.retirementDecayLocked`（§5.5）

#### 5.1.1 擬似式
```
function staminaBaseCapForAge(age):
  if age <= 50: テーブル参照
  else:          return 160 - (age - 50) * 2
```

### 5.2 体力ボーナス上限

- 一部の遊びをレベルアップさせると恒久的に体力上限がブーストされる。
- プロト段階では未実装。将来拡張のためのフィールド `player.staminaBonusCap` を 0 で初期化する。
- 実効体力上限 = `staminaBaseCap + staminaBonusCap`。

### 5.3 上限変化のタイミング
- **誕生日（年齢繰り上げ）時に再計算**する。新しい上限が低い場合、`stamina` は新上限までクランプ。
- 遊びによる体力ボーナス追加時も即時に再計算。

### 5.4 体力ゼロ時の挙動（ライフステージ別）

| ライフステージ | 対象年齢 | 挙動 |
|---|---|---|
| 保育園・幼稚園・小学校 | 1〜12 | **強制睡眠**：2時間の仮眠で体力を 30% 回復。時刻 +2h。続けて遊び可能 |
| 中学校・高校・大学・社会人 | 13〜65 | **強制終了**：その日は即就寝。翌朝の余剰時間が 0 になり、起床直後にコアタイムに入る |
| 老後 | 66〜100 | **強制終了**：その日は即就寝。体力ゼロ記録日として `player.depletedDays` に追加 |

#### 5.4.1 強制睡眠の詳細（1〜12歳）
- 発生場所：遊びの結果フェーズ（SPEC-003）で体力が 0 になった直後。
- 処理：
  1. 回復量 = `staminaCap * 0.30`（小数点以下切り捨て）
  2. `player.stamina` を回復量にセット
  3. 時刻を 2h 進める
  4. 余剰時間を 2h 減らす（ただし **§5.4.1a の吸収ルール** が優先）
  5. トースト「強制的にお昼寝した…体力が少し戻った」
  6. 余剰時間が 0 になった場合、自動で S5 就寝画面に遷移

#### 5.4.1a 強制睡眠のコアタイム吸収ルール（v2 追加・重要）

朝の遊び中に強制睡眠（2h 仮眠）が発動した場合、**睡眠時間の一部または全部をコアタイムに被せる** ことで、プレイヤーの夜の余剰時間を守る。

ルール：
- 仮眠 2h のうち、**コアタイム開始時刻以降に重なった分は「コアタイムが仮眠を吸収した」扱い** にし、夜の余剰時間から差し引かない。
- 仮眠終了時刻がコアタイム終了時刻を超えた場合は、**超過分のみ** 夜の余剰時間から差し引く。

計算式：
```
仮眠開始時刻   = player.clockHour（進める前）
仮眠終了時刻   = 仮眠開始時刻 + 2h
コアタイム開始 = coreTime.startHour
コアタイム終了 = coreTime.endHour

コアタイムと仮眠の重なり = max(0, min(仮眠終了, コアタイム終了) − max(仮眠開始, コアタイム開始))
仮眠でコアタイム外に消費した時間 = 2 − コアタイムとの重なり
超過分（コアタイム後に食い込んだ仮眠）= max(0, 仮眠終了 − コアタイム終了)

朝の余剰時間への影響 = 仮眠でコアタイム外に消費した時間 − 超過分
  = ただし朝の余剰は 2h を上限とする

夜の余剰時間への影響 = 超過分
```

イメージ：
```
ケース A: 朝 8:00 に体力 0 → 仮眠 8:00〜10:00 → コアタイム 9:00〜16:00 と 1h 重なる
  朝の余剰 -1h（8:00〜9:00 が仮眠）
  コアタイムが 9:00 に自動開始（コアタイム消化は通常どおり）
  夜の余剰 -0h（コアタイム時間内だけで仮眠完了） ← これが改修の要点

ケース B: 朝 7:00 に体力 0 → 仮眠 7:00〜9:00 → 全部コアタイム前
  朝の余剰 -2h
  コアタイム 9:00〜16:00 通常消化
  夜の余剰 -0h

ケース C: コアタイム中（12:00 で）仮眠発動 → 12:00〜14:00、全部コアタイム内
  仮眠がコアタイムを侵食しないので、特に夜の余剰への影響はない（プロト段階ではコアタイム中の遊びは発生しないので発生しないケース）

ケース D: コアタイム後 16:30 に体力 0 → 仮眠 16:30〜18:30
  朝の余剰 -0h
  夜の余剰 -2h（通常どおり）
```

#### 5.4.1b 吸収ルール発動時のコアタイム進行

- 仮眠によって時計が `coreTime.startHour` を超えた場合、その時点で通常の「朝の遊び終了 → コアタイムへ」の分岐が自動的に走る（SPEC-003 §5.8 core ctx）。
- プレイヤーはそのまま「🏫 保育園に行く」ボタンでコアタイム画面に進める。仮眠時点で時刻が 16:00 を超えることはほぼ無いが、もし超えた場合は `clockHour = coreTime.endHour` にクランプして `coreTimeDoneToday = true` を立て、夜の遊びへ遷移する。

#### 5.4.1c 計算の簡略化（プロト実装）

プロト段階では以下の簡略式で実装する：
```js
// 仮眠開始時刻
const napStart = player.clockHour + player.clockMinute / 60;
const napEnd   = napStart + 2;

// コアタイム情報
const stage = resolveLifeStage(player.age);
const core  = stage ? stage.coreTime : null;

// 1) 時刻を 2h 進める
player.clockHour   = Math.floor(napEnd);
player.clockMinute = Math.round((napEnd - Math.floor(napEnd)) * 60);

// 2) 朝の余剰（now < core.startHour の間のみ有効）から差し引くべき時間
let morningDecrease = 0;
if (core && napStart < core.startHour) {
  morningDecrease = Math.min(2, core.startHour - napStart);
}

// 3) コアタイム超過分（夜の余剰から差し引く）
let eveningDecrease = 0;
if (core && napEnd > core.endHour) {
  eveningDecrease = napEnd - core.endHour;
}

// 4) コアタイムがない（老後）または朝の余剰とも関係ないケース
if (!core) {
  // 老後は通常どおり余剰時間から 2h 差し引き
  player.spareHours = Math.max(0, player.spareHours - 2);
} else {
  // 朝の余剰を消費（朝の遊び時間中だった場合）
  player.spareHours = Math.max(0, player.spareHours - morningDecrease);
  // 夜の余剰分は後で coreTime 終了後に recomputeSpareHoursAfterCoreTime() を通して
  // 自動的に反映される。超過分がある場合は別途減算する。
  if (eveningDecrease > 0) {
    player._napOverflow = (player._napOverflow || 0) + eveningDecrease;
  }
}
```

`_napOverflow` は `recomputeSpareHoursAfterCoreTime()` で `spareHours` から差し引いて消費する（コアタイム消化後に一度だけ適用）。

#### 5.4.2 強制終了の詳細（13歳以上）
- 発生場所：遊びの結果フェーズで体力が 0 になった直後。
- 処理：
  1. トースト「倒れるように眠りに落ちた…」
  2. 即時に S5 就寝画面に遷移（プレイヤーに選択肢は与えず `normal` を自動選択）
  3. 翌日 `player.spareHours = 0` を強制セット
  4. 翌朝起床時、コアタイムに即入る（余剰時間での自由な朝が無くなる）

### 5.5 老後の翌年減衰緩和（生きがいルール）

- 老後（66歳以降）限定。
- **1年間（=プロトでは 120 日）で体力ゼロ記録日が `floor(120 / 4) = 30 日以上` ある場合、翌年の体力上限減衰（-2/年）がスキップ** される。
- 意味：「限界まで遊び切った年」は、翌年の老いが止まる。遊びの密度が長寿につながるメッセージ。
- 実装：
  1. 年齢繰り上げ時点で `player.depletedDaysThisYear` を集計。
  2. 30日以上なら `player.retirementDecayLocked = true` を立て、今年の減衰処理をスキップ。
  3. 次年度以降も同様の判定をその都度行う。
  4. 年末にカウンタをリセット。

### 5.6 プレイヤー状態スキーマ（拡張）

```
staminaCap            動的再計算される実効上限（base + bonus）
staminaBaseCap        年齢ベースの上限
staminaBonusCap       遊び・発見で追加される恒久ボーナス
depletedDays          体力ゼロを記録した日の配列（numeric day）
depletedDaysThisYear  今年の体力ゼロ記録日数（年齢繰り上げ時に集計）
retirementDecayLocked 老後減衰ロック（trueの年は -2/year が入らない）
_napOverflow          強制睡眠がコアタイム終了後まで食い込んだ時間（h）。
                      コアタイム消化時に夜の余剰から差し引かれ、即リセット。
                      （SPEC-019 §5.4.1a の吸収ルール用）
```

## 7. 想定される利用シナリオ

### 7.1 2歳で遊びすぎた
- 体力上限 15 の 2歳児が、鬼ごっこ（-20）を選ぶ → 体力 0 → **強制睡眠** → 体力 4（=15 × 0.30 切り捨て）へ回復、時刻 +2h。

### 7.2 30歳社会人で残業しすぎた
- 体力 220 から仕事・残業・ジムで 0 → **強制終了** → 即就寝 → 翌朝余剰時間 0 のまま コアタイムへ。

### 7.3 90歳で毎日限界まで遊ぶ
- 体力上限 80 の 90歳が、1年で 35 日の体力ゼロ記録 → 翌91歳の上限減衰をスキップ（上限 78 のまま）。

## 8. 例外・エッジケース

- 体力上限が変動して現在値が上限を超えた場合（例：誕生日で上限が減った）は即クランプ。
- 体力が負になる遊びの組合せ（複数のステータス消費イベント同時発火）は最終的に 0 でクランプ。
- プロト段階では保育園（1〜3歳）のみ実装。幼稚園以降は未実装（コアタイム未実装と同じ扱い）。
