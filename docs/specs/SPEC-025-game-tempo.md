# SPEC-025: ゲームテンポ設計（自動進行 + ステージ依存の時間流速）

| 項目 | 内容 |
|---|---|
| 仕様ID | SPEC-025 |
| 機能名 | ゲームテンポ設計（自動進行 + ライフステージ依存の時間流速） |
| 対応ファイル | `prototype/game.js` の `PASSION_PROFILES`, `tempoDaysPerTurn()`, `runAutoTurn()`, `autoAdvanceCoreTime()`, `autoAdvanceSleep()`, `renderDaySummary()` / `prototype/index.html` の `#screen-passion-profile`, `#screen-day-summary` |
| 関連仕様 | SPEC-001, SPEC-002, SPEC-003, SPEC-010, SPEC-011, SPEC-019, SPEC-020, SPEC-021, SPEC-022, SPEC-023, SPEC-024 |
| ステータス | Active（Option A **採用**。Option B はレビューで却下） |
| 最終更新 | 2026-04-18 |

> 本仕様は Issue #7「ゲームのテンポを良くしたい」に対応する。  
> 論点：1日×100年＝36,500 ターン、1日3判断とすると 10万タップ超でゲームテンポが悪い。  
> プレイヤーに **「人生全体」を 30 〜 60 分で体験させる** ためには、タップ数とテンポの最適化が必須。

---

## 1. 目的

- プレイヤーの **意思決定タップを 1 プレイあたり 200〜400 タップ** に圧縮する
- **「没頭＝自動で楽しい時間が過ぎる」** 感覚を演出し、プレイヤーは「節目」や「大事な選択」だけに介入
- **幼少期はゆっくり、社会人以降は速く** 時間が進むことで、人生のリアリティと時間感覚を再現
- 判断機会を **年齢フェーズに応じて最適な粒度** で提供する

## 2. 用語定義

- **情熱プロファイル（Passion Profile）**：プレイヤーが最初に（または節目で）設定する「遊びの好み」。例：公園派、インドア派、読書家、スポーツ派、クリエイター…
- **自動ターン（Auto Turn）**：プレイヤーの介入なしにゲームが 1 ターン（定義による）を消化する処理
- **介入イベント（Interruption Event）**：自動ターン中でも **プレイヤーのタップが必要** な出来事。例：Lv アップ、友人が増えた、解禁遊び出現、ライフステージ変化、体力ゼロ
- **時間流速（Tempo）**：1 ターンでゲーム内何日／何週間／何ヶ月進むかの単位
- **ハイライト（Highlight）**：自動ターンの後にまとめて表示される「この期間の出来事」のダイジェスト

## 3. 基本方針（Issue #7 のB+Cハイブリッド）

- 案C「自動進行・自動選択」を **全体の骨組み** として採用
- 案B「ライフステージ依存の時間流速」を **自動進行の粒度** に反映
- 案A「週単位」は週タームを基本粒度として保持するが、ステージによって 1〜30 日の可変単位にする

## 4. ライフステージ別の時間流速（Tempo Table）

`tempoDaysPerTurn(age)`：**1 回の自動ターンで経過する日数** を返す関数。年齢別：

| 年齢帯 | ライフステージ | 1 ターンの日数 | 1 年あたりのターン数 | 理由 |
|---|---|---|---|---|
| 1〜4 | 保育園 | **1 日** | 約120 | 最初の学びは丁寧に体験させる |
| 5〜6 | 幼稚園 | **2 日** | 約60 | 少しずつ加速 |
| 7〜12 | 小学校 | **3 日** | 約40 | 学校生活の繰り返し |
| 13〜15 | 中学校 | **7 日（1週間）** | 約17 | 週単位の部活動リズム |
| 16〜18 | 高校 | **7 日** | 約17 | 同上 |
| 19〜22 | 大学 | **14 日（2週間）** | 約8〜9 | 長期休みが多く、月の半分単位 |
| 23〜35 | 社会人（若手） | **30 日（1ヶ月）** | 約4 | 忙しくて毎日の記憶が薄い |
| 36〜50 | 社会人（中堅） | **30 日** | 約4 | |
| 51〜65 | 社会人（熟練） | **60 日（2ヶ月）** | 約2 | 毎年があっという間 |
| 66〜 | 老後 | **30 日** | 約4 | 再び時間を味わいたくなる |

### 4.1 人生を通算したターン数

上記テーブルで 1〜100 歳までを走破すると **約 1,100〜1,200 ターン** になる。
1 ターン 1 タップ（＋介入イベント 10〜20%）なら、**総タップ数 1,300〜1,500** 程度に収まる。
30〜60 分プレイで完走可能なペース。

### 4.2 プロト実装の段階的適用

- v1（保育園のみ実装済み）：年齢 1〜4 歳は `1 日 / ターン`（従来通り）
- v2（本仕様で保育園に自動進行を導入）：保育園も **1 日 / ターン** のまま、**自動遊び選択** のモードを提供
- 将来：幼稚園以降の実装時に Tempo Table を適用して、ターン粒度を拡げる

## 5. 情熱プロファイル（Passion Profile）

### 5.1 データ構造

```ts
type PassionProfile = {
  id: string;
  label: string;
  icon: string;
  description: string;
  preferredCategories: string[];  // SPEC-022 のカテゴリ ID を優先度順に列挙
  color: string;
};
```

### 5.2 プロト段階のプロファイル 4 種（保育園向け）

子どもが持つ「好みの芽生え」を 4 種類に要約。将来はステージごとに拡充する。

| ID | ラベル | 好む要素 |
|---|---|---|
| `outdoor_kid` | 🌳 外が好きな子 | `movement`, `outdoor_toy`, `nature`, `water` |
| `creator_kid` | 🎨 ものづくりが好きな子 | `crafting`, `imagination`, `music` |
| `bookworm_kid` | 📚 じっくり派 | `reading`, `literacy`, `sensitivity`, `science` |
| `social_kid` | 🤝 友達といる子 | `social_play`, `friendship`, `imagination` |

### 5.3 選択タイミング

- **初回起動時**：必須。選ばないとプレイ開始できない
- **ライフステージ切替時**：将来実装。プロフィールをアップデートできる
- **プロト段階**：保育園開始時に一度だけ選択

### 5.4 自動選択アルゴリズム

```
function pickAutoPlay(availablePlays, profile):
  // 1) スキル×好みカテゴリスコアを計算
  scored = availablePlays.map(play => ({
    play,
    score: computeScore(play, profile)
  }))

  // 2) スコア上位 3 つの中から重み付き抽選（常に同じ遊びにならないように）
  topK = sortDesc(scored)[:3]
  return weightedPick(topK)
```

#### 5.4.1 スコア式

```
score(play, profile) =
    (profile.preferredCategories ∩ play.categories).length × 10
  + averageSkillLv(play.categories) × 0.5      // 熟練した遊びをしたい性向
  + (play.unlockRequired ? 5 : 0)              // 解禁されたばかりの遊びは試したい
  + randomJitter(0, 2)                         // 少しだけ揺らぐ
```

### 5.5 「自分で選ぶ」モードとの切替

- 自動進行モード中でも、プレイヤーが **「自分で選ぶ」ボタン** を押せば従来の S2 遊び選択画面に戻る
- 自動進行 ↔ 手動進行 の切替は **ドック最左端のトグル** にする（Option A の実装）

## 6. 介入イベント（Interruption Event）

### 6.1 ポリシー

自動モードの本質は「テンポよく時間が進む」ことなので、**介入は最小限** にする。  
v2（PR #9 レビュー反映）では、従来プレイヤータップを要求していたコアタイム終了画面・就寝画面も **自動モード中はスキップ**。

### 6.2 自動モード中のスキップ対象（v2 追加）

| 場面 | 旧（v1） | 新（v2） |
|---|---|---|
| S6 コアタイム終了「家に帰る」 | プレイヤーのタップが必須 | **自動モード中は即閉じ→次ターンへ**（`autoAdvanceCoreTime()`） |
| S5 就寝「おやすみ」（1〜9歳 モード選択なし） | プレイヤーのタップが必須 | **自動モード中は即睡眠→翌朝へ**（`autoAdvanceSleep()`） |
| S3 遊びの描写 | プレイヤーのスキップが必要 | 自動 finalize（従来通り autoFinalizePlay） |

上記のスキップにより、**自動モードで 1 日を進めるのに必要なタップ数はゼロ**（介入イベントが発生しない日なら）。

### 6.3 それでも介入を要求する場面（介入のまま残す）

| イベント | 介入内容 |
|---|---|
| 新しい遊びの解禁 | 「◯◯を覚えた！」のモーダル（自動モードでも必須で見せる） |
| ライフステージ切替 | 「幼稚園に入園した」のモーダル |
| 体力ゼロ（強制睡眠 / 強制終了） | SPEC-019 の既存フロー（結果画面を見せる） |
| 友人増減（±2 以上の大きな変化） | モーダルで通知 |
| ランダムイベント（SPEC-004） | そのまま S4 表示 |
| **1 日の終わりに毎日到達する「日の終わりサマリ」** | **S10 を表示して必ずプレイヤーの意思を問う**（§7.2） |

### 6.4 手動モードへの復帰
- プレイヤーが手動モードにすると、上記スキップは一切行わず従来フローに戻る
- 日の終わりサマリの「手動に切り替え」ボタンからも復帰可能

## 7. サマリ（2 階層構造）

自動モード中は **2 種類のサマリ画面** を使い分ける。プレイヤーに「まとめて経験値が上がった手応え」を明示するため、**いずれも SPEC-021 §5.4a のゲージ差分アニメ** で表示する。

### 7.1 日の終わりサマリ（S10）v2 追加

毎日の就寝タイミングで **必ず 1 回** 表示される「その日のまとめ」画面。

#### 7.1.1 遷移タイミング
- 自動モード中：夜の余剰時間 0 または「今日おわる」自動化後に、**S5 就寝をスキップして代わりに S10 を表示**
- 手動モード中：従来の S5 就寝画面でモード選択 → S10 が挿入される

#### 7.1.2 表示項目
| セクション | 内容 |
|---|---|
| 期間 | 「N日目（2歳） / 夏」 |
| 遊んだ内容 | 遊び別にアイコン＋名前＋回数 |
| 原体験ゲージ | 5 カテゴリのうち増分があったもの。**`renderGaugeWithDelta` を使って差分ハイライト付き** で描画 |
| スキルゲージ | 伸びたスキルを **カテゴリごとにゲージ表示**（同上）|
| ステータス | 体力（回復後値）、ジョウネツ、友人数、所持金の差分 |
| 学んだこと | 新規解禁・発見イベントの 1 行メモ |

#### 7.1.3 フッター（3 択）
プレイヤーは「次にどこまでスキップするか」を選べる：

| ボタン | 動作 |
|---|---|
| `⏩ 週末までスキップ` | **その週の日曜日まで自動進行**（`daysUntilWeekend()` を使い、今が水曜なら残り4日、金曜なら残り2日、日曜なら翌週日曜まで7日進む）|
| `⏩ 明日の夜までスキップ` | 次の日の日の終わりサマリまで自動進行 |
| `🎮 手動に切り替え` | 自動モード解除、S2 遊び選択画面の手動モードに復帰 |

### 7.1.4 週末判定（Weekend）
- プロトでは 1 週間 = 月〜日の 7 日
- **週末 = その週の日曜日（fiscal week の最終日）**
- Day 1 = 2026-04-01（水）= その週の 3 日目
- `daysUntilWeekend(day)` = `(7 - dayOfWeek) mod 7` が 0 なら 7 を返し、今日が日曜でも次の日曜まで進む（連打時の退屈を避ける）
- ただし **既に週末サマリ（S9）が待っている時** は S10 でサマリを見せた後、自動的に週末サマリ S9 へ

### 7.2 ウィークリーサマリ / ハイライト（S9）

**週境界（日曜日の就寝後）** に 1 回だけ表示する集約画面。**S10 日サマリの後** に出る場合もあるため位置付けを明確化。

#### 7.2.0 v3：集計単位を 5 ターンから「週境界（日曜）」に変更
- 旧：5 ターンごと（＝保育園期は 5 日ごと）に出していた
- 新：**Fiscal week の境界で 1 回**（日曜日の就寝＝月曜日の朝に切り替わる直前）
- Fiscal week 定義は SPEC-001 §5.7 を参照（Day 1 = 2026-04-01 水曜、その週は Day 1-5 で締まる短縮週）
- スナップショットは週の最初の朝で取得し、週境界到達時に差分を表示

#### 7.2.1 v4：ハイライトバッファ（`_autoHighlight`）の更新タイミング
バグ報告「週末サマリの『今日の遊び』が『無し』になっていた」への対応として、以下を明文化する。
- `_autoHighlight.playsById` は **手動・自動のどちらのモードでもプレイ完了時に `+1`** すること
  - 手動：`finalizePlay()` 末尾で更新
  - 自動：`runAutoTurn()` 側で更新（従来どおり）
- `_autoHighlight` は **週ハイライト表示（S9）後にリセット**（`playsById = {}` / `turnCount = 0` / `dayStart = player.day` / `skillsBefore` と `expBefore` を現時点で取り直す）
- 起床時（`goChooseFromToday()`）に `initAutoHighlightIfNeeded()` を呼び、初回のみ `skillsBefore/expBefore/dayStart` を当日で初期化する

#### 7.2.2 v4：日サマリバッファ（`_daySnapshot`）の更新タイミング
バグ報告「1 日の終わりサマリの『今日の遊び』がその週の累計になっている」への対応。
- `_daySnapshot.playsById` は **その日の遊びのみ**を集計する。日付（`player.day`）が変わったタイミングで必ず `{}` に再初期化する。
- 手動モードでも **起床時に `initDaySnapshotIfNeeded()` を呼ぶ**。これは `dayAtStart !== player.day` の場合にのみ中身を作り直す（朝の複数回呼び出しは no-op）。
- `goChooseFromToday()` の冒頭で呼ぶことで、S4/S6 → S2 の遷移ごとに毎回発火しないが、日をまたいだ最初の到達時には必ず発火する。

#### 7.2.3 v5：S9 ウィークリーハイライト → 続ける押下時の遷移修正（ループバグ）
バグ報告「週末のハイライト→続けるを押すと日サマリに戻り、日サマリ→続けるを押すとハイライトに戻るループ」への対応。

- 旧実装：S9 の「続ける」で `goChooseFromToday()` を直接呼んでいた。しかし `player.day` が進んでいないため、朝の遊びが選べないまま S2 に戻り、そこで「今日おわる」を押すと S10 → S9 のループに陥っていた
- 新実装：**S9「続ける」は必ず `sleep("normal")` を呼んで翌日（月曜朝）に進める**
  - `sleep("normal")` → `nextDay()` → 起床時刻計算 → `goChooseFromToday()`
  - `_skipAfterWeekly` フラグは不要に（全経路で `sleep` 呼び出しに統一）
- フロー（正）：S10（日曜夜） → 続ける → S9（週末ハイライト） → 続ける → **月曜朝 S2** → （以下継続）

#### 7.2.1 表示項目
| セクション | 内容 |
|---|---|
| 期間 | 「N日目〜M日目（2歳）」 |
| 遊んだ回数 | 遊び別の合計回数（降順） |
| 原体験ゲージ | 5 カテゴリ分を **ゲージ表示**（差分ハイライト） |
| スキルゲージ | 伸びたスキルを **ゲージ表示**（差分ハイライト） |
| 発見 | 新規解禁・発見イベント |

#### 7.2.2 フッター
- `続ける`：次の 5 ターンへ継続
- `手動に切り替え`：S2 に戻る

### 7.3 2 階層サマリの狙い
- S10（日サマリ）：プレイヤーが **1 日単位で意思決定**（週末まで？明日まで？手動？）できる小さな区切り
- S9（週サマリ）：5 日分の積み上げを俯瞰して「長めの手応え」を感じられる区切り
- 両方を経由することで、プレイヤーが **飽き**・**方向感覚のズレ** を感じにくくなる

## 8. 選択肢の重要度による粒度切替（Option A の差別化）

- 自動ターンは **ほぼ全自動**
- プレイヤーが手動に切り替えると従来の S2 画面で 1 ターン介入
- 介入イベントは必ずプレイヤーを待つ

## 10. 採用方針（Option A）と Option B 却下の理由

Issue #7 に対して当初 2 パターンを検討した：

| 観点 | Option A：自動進行ダイジェスト型（**採用**） | Option B：週／月スキップ型（却下） |
|---|---|---|
| コア体験 | 1 ターンずつ自動で流れ、サマリで手応え | プレイヤーが「まとめて進める」ボタン押下 |
| プレイヤーのタップ | **画面下のフッターのみ（親指で操作完結）** | 画面中央上のボタン（指の移動が必要） |
| UX | 放置系の滑らかな没頭感 | 能動的操作だが親指の移動コスト |
| 採否 | ✅ 採用 | ❌ レビューで UX 劣位のため却下 |

### 10.1 採用理由（PR #9 レビューより）

> B案と比較してA案は押下するボタンが常に画面下にあるのが UX 上良かった。指を動かさずに操作ができる。

→ スマホ片手操作を前提としており、**親指が届く画面下エリアだけで全操作が完結** する Option A のほうが本作のコンセプトに合致する。

### 10.2 今後のロードマップ（Option A 前提）

- v1（マージ済み）：プロファイル選択／自動ターンループ／5 ターンハイライト
- **v2（本仕様・本ブランチ）**：S6 保育園・S5 就寝の自動スキップ／**S10 日の終わりサマリ**（3 択フッター）／S9 ハイライトのゲージ化
- v3（将来）：幼稚園以降の Tempo Table 適用、プロファイル切替画面

## 11. 想定される利用シナリオ（v2）

### 11.1 2歳プレイヤー 1 日目（すべて自動）
1. 初回：「🌳 外が好きな子」を選択
2. 自動モード ON で S2 に出現 → 自動で朝の遊びが 2 ターン流れる（砂場→絵本）
3. 09:00 到達 → **保育園 S6 を自動スキップ**（画面遷移のみ。プレイヤーはタップ不要）
4. 保育園発見：🎢 大きな滑り台 → 介入モーダルで 1 タップ「続ける」
5. 16:00 → 夜の遊び 2 ターン（粘土→ごっこ）
6. 18:00 → **就寝 S5 を自動スキップ**
7. **S10 日の終わりサマリ** が表示される（ゲージで原体験・スキル増分を視覚化）
8. プレイヤーは「⏩ 明日の夜までスキップ」「⏩ 週末までスキップ」「🎮 手動」から選択

### 11.2 「週末までスキップ」選択後
- 連続で 6 日分を自動進行
- その間、新発見があればモーダルで 1 タップずつ介入
- 5 日目で S9 ウィークリーハイライトが出現し、6 日目でもう一度 S10 が出る
- ※ 介入の都度、プレイヤーが手動に切り替える選択肢は S10 から常に取れる

## 12. 例外・エッジケース

- **プロファイル未選択**：初回起動時は強制的に S8 選択画面を表示。キャンセル不可
- **自動選択の結果、すべてロック**：手動モードに自動で切替、ドックにも「遊べるものがない」メッセージ
- **介入イベントが連続発生**：各イベントをキューイングし、1 つずつ順番にモーダル表示
- **強制睡眠（体力 0）**：SPEC-019 の既存フローに戻し、S10 はその日の分は **出さず** S5 系 UI を経由する（将来併合検討）
