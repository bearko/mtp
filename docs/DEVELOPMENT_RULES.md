# 開発ルール / Development Rules

> 本プロジェクトは **仕様駆動開発（Spec-Driven Development）** を徹底する。
> 本ドキュメントは **開発を行うすべての作業の前後で必ず参照** し、違反している場合は必ず是正すること。

---

## 0. 大原則

1. **仕様書がない機能は実装しない。**
2. **仕様書に書かれていない挙動を実装しない。**
3. **コードを変更したら、必ず対応する仕様書を同じPRで更新する。**
4. **仕様書はコードを読めない人でも挙動を理解できる粒度で書く。**

---

## 1. ディレクトリ構成

ドキュメントは **画面設計書（Screens）** と **機能仕様書（Specs）** の 2 カテゴリに明確に分ける。

```
docs/
├─ DEVELOPMENT_RULES.md       ← 本ルール
├─ screens/                   ← 画面設計書（How it looks）
│   ├─ INDEX.md                 画面索引
│   └─ SCREEN-DESIGN.md         全画面のワイヤーフレーム・IPO・遷移
└─ specs/                     ← 機能仕様書（How it behaves）
    ├─ SPEC-INDEX.md            仕様書の索引
    ├─ SPEC-001-life-stage.md
    ├─ SPEC-002-play-selection.md
    └─ ...                      1機能=1ファイル
```

### 1.1 画面設計書 vs 機能仕様書 vs ゲームフロー図

| 観点 | 画面設計書（Screens） | ゲームフロー図 | 機能仕様書（Specs） |
|---|---|---|---|
| 答えるもの | **どう見えるか・どう遷移するか** | **いつ・どんな順番で・どんな分岐で起こるか** | **何が起こるか・どう計算するか** |
| 代表要素 | ワイヤーフレーム、画面ID、HUD レイアウト | Mermaid 図、ランダム発火・繰り返し・分岐 | データモデル、計算式、閾値、挙動フロー |
| 粒度 | 画面単位（S0, S2, …）と全体俯瞰 | フロー単位（コアループ / イベント系統） | 機能単位（SPEC-001, SPEC-002, …） |
| ファイル | `docs/screens/SCREEN-DESIGN.md`, `docs/screens/INDEX.md` | `docs/screens/GAME-FLOW.md` | `docs/specs/SPEC-XXX-*.md` |
| 相互参照 | 対応する `SPEC-XXX` を各セクション冒頭で明記 | コアループの全体像で SPEC ID を参照 | `## 6. UIへの反映` で画面 ID `Sn` を参照 |

- **1 機能 = 1 仕様書**。ファイル名は `SPEC-{連番}-{機能名}.md`。
- 仕様書の索引は `docs/specs/SPEC-INDEX.md`、画面設計書の索引は `docs/screens/INDEX.md` に一覧化する。
- **ゲームフロー図** は SPEC を横断する「動的な一覧」。画面追加だけでなく、新しいランダム発火・分岐・モード（自動 / スキップ / 病気など）を追加する際は必ず更新する。

## 2. 仕様書のフォーマット

全ての仕様書は以下のテンプレートに沿う。

```markdown
# SPEC-XXX: 機能名

| 項目 | 内容 |
|---|---|
| 仕様ID | SPEC-XXX |
| 機能名 | ○○ |
| 対応ファイル | `prototype/game.js` (関数 `foo()`), `prototype/index.html` (#screen-yyy) |
| 関連仕様 | SPEC-XXX, SPEC-YYY |
| ステータス | Draft / Active / Deprecated |
| 最終更新 | YYYY-MM-DD |

## 1. 目的
この機能が存在する理由を、プレイヤーから見た価値で1〜3行で説明する。

## 2. 用語定義
使用する用語を列挙し、意味のぶれをなくす。

## 3. 入力（Input）
この機能が動作するために必要な入力値・状態を箇条書きで列挙。

## 4. 出力（Output）
この機能が返すもの、画面に表示される結果、永続化される値。

## 5. 挙動（Behavior）
コードが読めない人でも挙動を追えるように、
- 手順（ステップ1、ステップ2…）
- 条件分岐（もし○○なら△△）
- 数式（計算式があるもの）
を平易な日本語で記述する。

## 6. UIへの反映
どの画面で、どの要素に表示されるかを画面設計書のIDで参照する。

## 7. 想定される利用シナリオ（例）
代表的な使われ方を1〜3例、具体的な数値で示す。

## 8. 例外・エッジケース
想定される例外とその扱い。

## 9. 未決事項・TODO
```

## 3. コード側の規約

### 3.1 仕様書リンクの注釈必須
- 仕様書に紐づく関数・モジュールには、**ファイル先頭または関数直上** に以下のコメントを必ず入れる：

```js
/**
 * @spec docs/specs/SPEC-002-play-selection.md
 * @summary 遊び候補のフィルタリング処理
 */
function isPlayAvailable(play) { ... }
```

- 仕様書に存在しないロジックを実装するときは、**先に仕様書を作成する**。

### 3.2 画面IDとコードの対応
- HTMLの画面セクションには、画面IDと画面設計書／仕様書のリンクをコメントで明記：

```html
<!--
  @screen S2 遊びを選ぶ
  @screens docs/screens/SCREEN-DESIGN.md  画面設計
  @spec    docs/specs/SPEC-002-play-selection.md
-->
<section id="screen-choose">...</section>
```

- `@screens` は画面設計書へのリンク、`@spec` は機能仕様書へのリンクを指す。画面に関わる変更ではその両方を参照し、必要なら両方更新する。

### 3.3 変更禁止事項の注釈
- 仕様書で明示的に定められた定数・アルゴリズムを変更する場合、**仕様書を先に変更**し、コード内で `@spec-version` を更新すること。

---

## 4. 開発フロー（必ず順守）

1. **変更要求を理解する**
2. **既存の仕様書を読む**（対応する SPEC-XXX を確認。なければ新規作成する）
3. **仕様書を先に更新する**（Spec First）
4. **画面設計書を更新する**（UI変更を伴う場合）
5. **コードを変更する**（@spec 注釈を必ず追加/更新）
6. **動作確認する**
7. **仕様書 / 画面設計書 / コード に矛盾がないかセルフレビューする**
8. **コミット & PR 更新**

> **⚠ ショートカット禁止**：「小さい修正だから仕様書は後で」は全面的に禁止。

---

## 5. コミットメッセージ規約

```
<type>(<scope>): <summary>

<body>

Spec: docs/specs/SPEC-XXX-xxx.md (updated|added|none)
```

- `type`: `feat` / `fix` / `refactor` / `docs` / `chore`
- `scope`: `prototype`, `spec`, `design` など
- 仕様書更新がないコード変更は **原則禁止**。やむを得ない場合は body にその理由を明記。

## 6. セルフレビューチェックリスト（PR 前に必ず）

### 6.1 機能仕様（必須）

- [ ] 変更した機能に対応する **SPEC-XXX 仕様書** があるか？（無ければ新規作成）
- [ ] その SPEC の「最終更新」日を **今日の日付** に更新したか？
- [ ] その SPEC の「対応ファイル」欄に新規関数・新規 DOM ID を追記したか？
- [ ] その SPEC の「関連仕様」欄を、**双方向**（自分が参照する側 / される側の両方）に更新したか？
- [ ] 仕様書の文面だけで「何が起きるか」が第三者に伝わるか？
- [ ] **Why not（採用しなかった選択肢の理由）** を 2 〜 5 件記載したか？

### 6.2 横断ドキュメント（**漏れやすい・要注意**）

> 「自分が触った SPEC は更新したが、他のドキュメントを忘れる」のが最も典型的なバグ源。
> 以下は **画面 ID・遷移先・グローバル状態・新規イベント・新規モード** のいずれかを変更したら必ず確認する。

- [ ] **`docs/screens/INDEX.md`**：画面 ID 一覧表は最新か？（新規画面 / オーバーレイ / モーダル ID を追加したか）
- [ ] **`docs/screens/SCREEN-DESIGN.md`**：画面 ID 一覧と遷移図、データモデル `PlayerState`、オーバーレイ z-index は最新か？
- [ ] **`docs/screens/GAME-FLOW.md`**：Mermaid フロー図に新しい分岐 / ランダム発火 / モードを追記したか？
  - 特に **発火タイミング** が変わるとき（朝 / 遊び後 / 累計達成 / 場所到着）は §3 の 4 系統表を更新
  - 自動モード / スキップ / チュートリアルなどの **特殊フロー** に影響するときは §4 / §5 / §8 を更新
- [ ] **`docs/specs/SPEC-INDEX.md`**：新規 SPEC を追加したら一覧表に行追加。既存 SPEC の最終更新日も更新。
- [ ] **`docs/DEVELOPMENT_RULES.md`**：開発ルール自体に影響するとき（新ジャンルの SPEC・新カテゴリのドキュメントを追加したとき）は §1 / §6.2 を更新。
- [ ] **`prototype/index.html`** の HTML コメントの `@screen` / `@spec` リンクは最新か？
- [ ] **`prototype/game.js`** の関数 JSDoc の `@spec` 注釈は最新か？

### 6.3 横断的に影響する変更類型と更新先

| 変更類型 | INDEX | SCREEN-DESIGN | GAME-FLOW | SPEC-INDEX | DEVELOPMENT_RULES |
|---|:-:|:-:|:-:|:-:|:-:|
| 新画面追加（`screen-*`） | ✅ | ✅ | ✅ | ✅（新 SPEC） | – |
| 新オーバーレイ追加（`#xxx-overlay`） | ✅ | ✅ | ✅ | ✅（新 SPEC） | – |
| 既存画面の遷移先変更 | – | ✅ | ✅ | ✅（該当 SPEC） | – |
| 新ランダム発火イベント | – | – | ✅（§3 系統表） | ✅（新 or 既存 SPEC） | – |
| 新モード（自動 / スキップ / 病気） | – | ✅（§1.4） | ✅（§4 / §5 / §8） | ✅（新 SPEC） | – |
| 新グローバル `player.*` フィールド | – | ✅（§6 データモデル） | – | ✅（該当 SPEC） | – |
| 新カテゴリの SPEC 群 | – | – | – | ✅ | ✅（§1） |
| ドキュメント運用ルール変更 | – | – | – | – | ✅ |

### 6.4 PR 本文の最低要件

- [ ] 変更した SPEC 一覧（追加 / 更新 / 廃止）を本文に列挙
- [ ] 横断ドキュメント（GAME-FLOW / SCREEN-DESIGN / INDEX）の更新可否を明記（更新が不要な場合はその理由を 1 行で）
- [ ] テスト追加 / 全テスト pass の確認
- [ ] **Why not** が SPEC に書かれている場合、PR 本文にも要約を転記

> **PR テンプレート**: `.github/PULL_REQUEST_TEMPLATE.md` がある場合はそれを使う。チェックリストが UI 上で表示される。
