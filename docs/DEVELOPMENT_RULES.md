# 開発ルール / Development Rules

> 本プロジェクトは **仕様駆動開発（Spec-Driven Development）** を徹底する。
> 本ドキュメントは **開発を行うすべての作業の前後で必ず参照** し、違反している場合は必ず是正すること。

---

## 0. 大原則

1. **仕様書がない機能は実装しない。**
2. **仕様書に書かれていない挙動を実装しない。**
3. **コードを変更したら、必ず対応する仕様書を同じPRで更新する。**
4. **仕様書はコードを読めない人でも挙動を理解できる粒度で書く。**

---

## 1. ディレクトリ構成

```
docs/
├─ DEVELOPMENT_RULES.md       ← 本ルール
├─ screen-design.md           ← 画面設計書（全画面のワイヤーフレーム・IPO・遷移）
└─ specs/                     ← 機能別仕様書
    ├─ SPEC-INDEX.md          ← 仕様書の索引
    ├─ SPEC-001-life-stage.md
    ├─ SPEC-002-play-selection.md
    ├─ SPEC-003-play-execution.md
    ├─ SPEC-004-random-event.md
    ├─ SPEC-005-parameter.md
    ├─ SPEC-006-bio-rhythm.md
    ├─ SPEC-007-friends.md
    └─ SPEC-008-sleep.md
```

- **1機能 = 1仕様書**。ファイル名は `SPEC-{連番}-{機能名}.md`。
- 仕様書の索引は `docs/specs/SPEC-INDEX.md` に一覧化する。

---

## 2. 仕様書のフォーマット

全ての仕様書は以下のテンプレートに沿う。

```markdown
# SPEC-XXX: 機能名

| 項目 | 内容 |
|---|---|
| 仕様ID | SPEC-XXX |
| 機能名 | ○○ |
| 対応ファイル | `prototype/game.js` (関数 `foo()`), `prototype/index.html` (#screen-yyy) |
| 関連仕様 | SPEC-XXX, SPEC-YYY |
| ステータス | Draft / Active / Deprecated |
| 最終更新 | YYYY-MM-DD |

## 1. 目的
この機能が存在する理由を、プレイヤーから見た価値で1〜3行で説明する。

## 2. 用語定義
使用する用語を列挙し、意味のぶれをなくす。

## 3. 入力（Input）
この機能が動作するために必要な入力値・状態を箇条書きで列挙。

## 4. 出力（Output）
この機能が返すもの、画面に表示される結果、永続化される値。

## 5. 挙動（Behavior）
コードが読めない人でも挙動を追えるように、
- 手順（ステップ1、ステップ2…）
- 条件分岐（もし○○なら△△）
- 数式（計算式があるもの）
を平易な日本語で記述する。

## 6. UIへの反映
どの画面で、どの要素に表示されるかを画面設計書のIDで参照する。

## 7. 想定される利用シナリオ（例）
代表的な使われ方を1〜3例、具体的な数値で示す。

## 8. 例外・エッジケース
想定される例外とその扱い。

## 9. 未決事項・TODO
```

## 3. コード側の規約

### 3.1 仕様書リンクの注釈必須
- 仕様書に紐づく関数・モジュールには、**ファイル先頭または関数直上** に以下のコメントを必ず入れる：

```js
/**
 * @spec docs/specs/SPEC-002-play-selection.md
 * @summary 遊び候補のフィルタリング処理
 */
function isPlayAvailable(play) { ... }
```

- 仕様書に存在しないロジックを実装するときは、**先に仕様書を作成する**。

### 3.2 画面IDとコードの対応
- HTMLの画面セクションには、画面IDと仕様書IDをコメントで明記：

```html
<!--
  @screen S1 起床
  @spec   docs/screen-design.md#s1-起床画面
-->
<section id="screen-wakeup">...</section>
```

### 3.3 変更禁止事項の注釈
- 仕様書で明示的に定められた定数・アルゴリズムを変更する場合、**仕様書を先に変更**し、コード内で `@spec-version` を更新すること。

---

## 4. 開発フロー（必ず順守）

1. **変更要求を理解する**
2. **既存の仕様書を読む**（対応する SPEC-XXX を確認。なければ新規作成する）
3. **仕様書を先に更新する**（Spec First）
4. **画面設計書を更新する**（UI変更を伴う場合）
5. **コードを変更する**（@spec 注釈を必ず追加/更新）
6. **動作確認する**
7. **仕様書 / 画面設計書 / コード に矛盾がないかセルフレビューする**
8. **コミット & PR 更新**

> **⚠ ショートカット禁止**：「小さい修正だから仕様書は後で」は全面的に禁止。

---

## 5. コミットメッセージ規約

```
<type>(<scope>): <summary>

<body>

Spec: docs/specs/SPEC-XXX-xxx.md (updated|added|none)
```

- `type`: `feat` / `fix` / `refactor` / `docs` / `chore`
- `scope`: `prototype`, `spec`, `design` など
- 仕様書更新がないコード変更は **原則禁止**。やむを得ない場合は body にその理由を明記。

---

## 6. セルフレビューチェックリスト（PR前に必ず）

- [ ] 変更した機能に対応する仕様書があるか？
- [ ] 仕様書の「最終更新」日を更新したか？
- [ ] 画面を変更したなら `docs/screen-design.md` も更新したか？
- [ ] 新規関数・画面に `@spec` 注釈を入れたか？
- [ ] 仕様書の文面だけで「何が起きるか」が第三者に伝わるか？
- [ ] PR本文に変更した SPEC 一覧を記載したか？

---

- 仕様書のバージョニング（`@spec-version` に基づくCI）
