## 1. 目的

全画面・全コンポーネントにわたるビジュアル言語と実装ルールを一箇所に集約する。  
個別 SPEC がUIの「何を表示するか」を定める一方で、本 SPEC は「どう見せるか」の根拠と制約を定める。  
今後のUI変更・新画面追加の際に本 SPEC を参照することで、デザインの一貫性を保つ。

---

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| **デザイン憲章** | ゲームの体験原則をUIに翻訳した不変のルール群 |
| **デザインシステム** | 色・タイポグラフィ・コンポーネントのトークン定義 |
| **ゴーストプレビュー** | 選択前にコスト・効果を半透明で重ね表示する手法（SPEC-034 §5.3 参照）|
| **3列HUDグリッド** | 左（ステージ/年齢/残時間）・中央（日付/リズム/体力）・右（金/友人/場所）の配置（SPEC-034 §4.1 参照）|
| **素養カード** | 5素養（身体/知性/感性/社交/情熱）を1行ずつ表示するカードコンポーネント |
| **グレード** | 素養値を7段階（A〜G）に変換した可視化指標 |

---

## 3. 入力（Input）

- `player.soyou.*`（5素養の数値）
- `player.stamina` / `player.staminaCap`
- `player.bioRhythm`（リズム絵文字変換に使用）
- `player.spareHours`（残り時間表示）
- `player.money` / `player.friends` / `player.location`
- 選択中の遊びオブジェクト（ゴーストプレビュー時）

---

## 4. 出力（Output）

- 画面全体の統一されたビジュアルトーン
- HUD・素養カード・ドック・結果画面・連絡帳の各コンポーネント
- ゴーストプレビュー状態（選択中のコスト/効果の重ね表示）

---

## 5. 挙動（Behavior）

### 5.1 デザイン憲章（16原則）

#### カテゴリ①：体験原則

1. **30秒ループ原則**  
   1ターンは30秒以内で完結する。タップ数は常に最小化し、スキマ時間に自然に手が伸びる軽快さを最優先する。1アクション = 1インテント。

2. **情報は「見る」で把握する**  
   HUDの状態は色・ゲージ・グレードのみで瞬時に判断できること。長いテキストを読まないと状態が分からない設計を禁止する。

3. **ゴーストプレビュー原則**  
   遊びを選択する前に、必ずコスト（体力・時間）と効果（素養上昇量・グレード変化）を画面上で可視化する。「選んで後悔」の体験を設計上排除する。

4. **音なし完結原則**  
   通勤・電車プレイを想定し、SEに依存せずに結果の種別（獲得/コスト/イベント）が色と記号だけで伝わること。

#### カテゴリ②：ビジュアル言語

5. **ダークパープル基調**  
   ベースカラーは深夜ダークパープル（`#0f0c1a`）。白背景は使用しない。明度の高いアクセントカラーで重要情報を際立たせる。

6. **セマンティックカラーの固定**  
   - 獲得・上昇 → グリーン `#4ade80`
   - コスト・消費 → レッド `#f87171`
   - カネ・時間残 → ゴールド `#f5c842`
   - 主軸アクセント（HUDラベル/CTA/グレード） → パープル `#a78bfa`
   - セカンダリCTA（次を選ぶ/1h休む） → スカイ `#38bdf8`  
   この対応を全画面で一貫させる。新しい文脈に色を割り当てる場合は本 SPEC を更新してから実装する。

7. **グレードは色＋文字で即時判断**  
   A（金）/ B（水）/ C（緑）/ D（橙）/ E（灰）/ F（紫灰）/ G（最低）の配色を固定する。数値は補助情報として小さく添える。色だけで良否が判断できること。

8. **季節アクセント**  
   季節ごとに背景グラデーションのアクセントを変化させて空気感を演出する（実装は将来フェーズ。春=桜ピンク/夏=空青/秋=橙/冬=紺）。

#### カテゴリ③：画面別ルール

9. **HUDは純機能UI**  
   HUDはゲームHUD的な情報密度で設計し、世界観演出は行わない。3列グリッド（2fr 3fr 2fr）を崩さない。

10. **S3結果・S10連絡帳は絵本テイスト**  
    演出系画面は温かみを持たせる。結果コメントは子どもの一人称・斜体で統一。連絡帳ヘッダーは「📓 連絡帳 — 今日のきろく」形式で世界観を強調する。

11. **ドックは全候補を可視化する**  
    遊びボタンは横スクロールで全候補をスクロールして閲覧できること。「見えない選択肢」は作らない。ツリーアイコンは右端に固定配置する。

12. **モーダルはフロー遮断しない**  
    オーバーレイのz-index管理はSCREEN-DESIGN.md §1.4に従い厳格に管理する。モーダル表示中にゲームフローのタイマーが進行しないこと。

#### カテゴリ④：コンポーネント規約

13. **素養カードは5列grid**  
    `icon | label | grade | delta | value` の5列構成を固定する。deltaセルと実数値セルを分離し、重ね表示を禁止する（SPEC-034 §6.3 教訓）。空deltaは空spanで列位置を固定する。

14. **体力ゲージは中央エリアの主役**  
    最低高さ6px。3段階カラー：緑（≥50%）/ 黄（20〜49%）/ 赤（<20%）。ゴーストプレビュー時は右端から半透明レッドで消費量を重ねる。

15. **null ガード必須**  
    UIリデザインでDOMを削除・追加した際は、`byId()` 参照箇所を全検索してnullガードを入れる（SPEC-034 §6.4 教訓）。`byId(x)?.hidden = true` の形式を推奨する。

16. **フォントはM PLUS Rounded 1cのみ**  
    見出し・数値は `font-weight: 900`、ラベルは `700`、本文は `400` の3ウェイト構成。最小フォントサイズは補助テキストの7px。それ未満は使用しない。

## 7. 想定される利用シナリオ（例）

### シナリオ1：新しい遊びボタンを追加する

1. 遊びの素養獲得種別を確認する（身体/知性/感性/社交/情熱）
2. ゴーストプレビューで当該素養カードの `delta` セルに `(+N)` を表示する
3. コストカラーは `--c-red` で体力ゲージゴーストを右端に重ねる
4. ボタンの選択状態は `border-color: var(--c-purple)` に変更する

### シナリオ2：新しいライフステージ（小学生）の画面を追加する

1. HUDグリッドの3列比率（`2fr 3fr 2fr`）は維持する
2. 所持金は小学生以降から表示（幼少期は非表示のルールを引き継ぐ）
3. ドックに表示する遊び候補の絵文字・名前・コストの形式を統一する
4. 結果コメントは小学生の一人称・斜体で記述する

### シナリオ3：新しい統計チップを追加する（例：体力チップ）

1. チップの背景は `rgba({R},{G},{B},0.12)`、テキストは対応するアクセントカラーとする
2. 現行チップ（身体=緑/知性=水/感性=ピンク/社交=金）と色が重複しないか確認する
3. チップの `font-size` は7px・`border-radius` は4pxを維持する

## 8. 例外・エッジケース

| 状況 | 扱い |
|---|---|
| 幼少期（age ≤ 6）の所持金 | HUD右エリアで非表示（DOM削除ではなく `hidden` 属性）|
| 幼少期のリズム | 常に「🤩 絶好調」固定（SPEC-034 §4.3 準拠）|
| delta が 0 の素養カード | deltaセルを空spanで固定し、グリッド崩れを防ぐ |
| 体力ゲージが0 | `sta-low`（赤）クラスを適用し、100%幅ではなく0%で表示 |
| 360px未満の画面幅 | HUD各エリアの2行目以降のラベルを省略（`text-overflow: ellipsis`）|
| 自動モード起動時のDOM参照 | 旧要素（`#wakeup-header` 等）は `hidden` 化のみ。null ガード必須（SPEC-034 §6.4）|

## Why not（採用しなかった選択肢）

1. **白背景 + カラーカード方式**  
   → 通勤・暗所での使用が多いスマホゲームに向かない。目への負荷が高く、没入感が低下する。ダークパープル基調を採用。

2. **体力をハート型アイコンの個数で表現**  
   → コスト計算が直感的にできない（「ハート3つ減る」より「30%ゲージが削れる」の方が視覚的に分かりやすい）。ゲージバー方式を採用。

3. **グレードを数値（0〜100）のみで表示**  
   → 「82」と「83」の差は瞬時に判断できないが「CとC+」なら分かる。A〜Gのグレード文字 + 色で補助する方式を採用。

4. **deltaを実数値の上に `position: absolute` で重ねる**  
   → 「82」の上に「+10」が重なって読めない問題が発生（SPEC-034 §6.3で実際に発生したバグ）。5列gridで並列表示する方式を採用。

5. **フォントにシステムフォントを使用**  
   → 「カネとジカンとジョウネツと」の柔らかく親しみやすい世界観にそぐわない。M PLUS Rounded 1c（丸ゴシック系）を採用。
