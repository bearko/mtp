## 1. 目的

本憲章は、本作の UI が迷ったときに戻る **判断基準** を定義する。

本作は「人生を効率よく攻略するゲーム」ではなく、**遊ぶこと・育つこと・見守られることの価値を思い出すゲーム**である。UI はプレイヤーに数値を管理させるためではなく、日々の選択が「思い出」として積み重なる感覚を支える。

---

## 2. デザインビジョン

> **小さな成長を、大切な人と一緒に見つける UI。**

プレイヤーが画面を見たとき、最初に感じるべきものは「攻略情報の量」ではなく、以下の 3 つである。

1. **今日はどんな日か**がすぐわかる
2. **次に何をすればよいか**が迷わずわかる
3. **何ができるようになったか**が嬉しく伝わる

---

## 3. 体験原則

### 3.1 比較しない

幼少期・幼稚園期の UI では、他人との順位・偏差値・上位率・ランキングを出さない。表示する比較は **昨日までの自分** と **今の自分** に限る。

### 3.2 数字よりエピソードを前に出す

素養値・体力・時間はゲーム判断に必要だが、感情の主役ではない。結果画面・連絡帳・ミッション達成では、数値差分より先に「何が起きたか」「誰が喜んだか」を伝える。

### 3.3 成長は 4 つの瞬間で見せる

SPEC-048 §12.1 の 4 軸を UI 表現の中心にする。

1. 今までできなかったことができるようになる
2. できたことを他の人に認めてもらう
3. 自分の成長を親が喜んでくれる
4. 自分のことで両親が褒められている

### 3.4 片手で、30 秒で、音なしでわかる

README §8 のスマホ前提を継承する。主要な 1 ターン操作は、音がなくても理解でき、片手親指で完結し、1 ターン 30 秒以内に進められることを目指す。

### 3.5 タップ数を減らしても、見どころは減らさない

自動モード・スキップ・統合画面はテンポ改善のために使う。ただし、成長・達成・発見・親や先生の反応などの「記憶に残る瞬間」は必ず視覚的に強調する。

---

## 4. UI 判断基準

迷った場合は、以下の優先順位で判断する。

1. **プレイヤーが次に押す場所が明確か**
2. **ゲーム内の現在状態が 3 秒以内に読めるか**
3. **成長・発見・達成が感情として伝わるか**
4. **幼少期の世界観を壊す表現になっていないか**
5. **既存の画面 ID / SPEC / デザインシステムと矛盾しないか**

---

## 5. 言葉の原則

### 5.1 使いたい言葉

- できた
- やってみる
- もうすぐ
- だいすき
- 思い出
- 今日はこんな日
- お母さん / お父さん / 先生が見ている

### 5.2 避けたい言葉

- 効率
- ノルマ
- 周回必須
- ランキング
- 偏差値
- 上位 X %
- 失敗
- 最適解

ただし、中学校以降の競技・受験・仕事フェーズでは、競争や評価の語彙を段階的に解禁してよい。

---

## 6. Why not

### 6.1 Why not: 攻略ダッシュボード中心

詳細なグラフや KPI を前面に出すと、プレイヤーは「人生を最適化する」視点になりやすい。本作の核は「遊びを選んだ日々が思い出になる」ことなので、分析画面は必要最小限に留める。

### 6.2 Why not: ソーシャル比較

他プレイヤーの平均やランキングは短期的な動機になるが、幼少期の「どの子も豊かに育つ」というメッセージを損ねる。比較は中学校以降の競技・受験など、世界観上自然な場面に限定する。

### 6.3 Why not: 汎用 RPG 風の重厚 UI

本作はバトル RPG ではなく、日常と人生のシミュレーションである。黒背景・金属質・過剰なエフェクトより、紙・絵本・連絡帳・アルバムの質感を優先する。

---

## 7. 運用

- UI 変更を伴う SPEC は、[`DESIGN-SYSTEM.md`](./DESIGN-SYSTEM.md) のトークン・コンポーネント方針に合わせる。
- デザイン方針そのものを変更する場合は、本憲章を更新し、関連する画面設計 / SPEC にリンクを追加する。
- 新しい UI パターンを作る前に、既存の「HUD」「カード」「モーダル」「連絡帳」「ドック」で表現できないか確認する。
